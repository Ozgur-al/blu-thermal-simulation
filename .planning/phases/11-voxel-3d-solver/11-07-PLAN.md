Add powered-block contact diagnostics and block geometry validation warnings.

Purpose: Users currently get no feedback about whether LED blocks actually contact their intended thermal path. The solver silently assigns materials via last-defined-wins, and a mis-ordered block list or missing contact can produce incorrect results with no warning. These diagnostics make thermal path issues visible before and after solving.

Output: A solver-side diagnostic function that reports neighboring materials and shared face area for each powered block, and an editor-side validation method that warns about common geometry issues.

```python
def diagnose_powered_block_contacts(
    project: VoxelProject,
) -> list[dict]:
    """Report neighboring materials and shared face area for each powered block.

    Returns a list of dicts, one per powered block:
    {
        "block_name": str,
        "power_w": float,
        "neighbors": [
            {"material": str, "face_area_m2": float, "direction": str},
            ...
        ]
    }

    Direction is one of: "+x", "-x", "+y", "-y", "+z", "-z".
    Only reports contacts with non-air materials, plus a single aggregate "air" entry
    if any air contact exists.
    """
```

Implementation approach:
1. Build conformal mesh and material grid (reuse `build_conformal_mesh` and `assign_voxel_materials`).
2. For each powered block (power_w > 0):
   a. Find the block's voxel extent (ix_min:ix_max, iy_min:iy_max, iz_min:iz_max) using cell-center containment.
   b. For each of 6 faces, identify the neighbor voxels just outside the block boundary. For example, for the +z face (top), the neighbor voxels are at iz=iz_max (one layer above the block's top voxels).
   c. Read material names from `material_grid` for those neighbor voxels.
   d. Sum face areas per unique material per direction.
   e. Collect into the return structure.

Use vectorized NumPy for the face-area sums (meshgrid of boundary indices, then group by material using `np.unique` or dict accumulation).

Also add a `logger.info()` call at the end of `build_voxel_network()` that calls this diagnostic and logs a summary line per powered block, e.g.:
`"Block 'LED_1' (0.5W): +z -> Aluminum (2.4e-5 m2), -z -> Air (2.4e-5 m2)"`

<task type="auto">
  <name>Task 2: Add block geometry validation warnings in block editor</name>
  <files>thermal_sim/ui/block_editor.py</files>
  <action>
Add a public method `validate_blocks(self) -> list[str]` to `BlockEditorWidget` that reads the current block table and returns a list of warning strings. This method performs geometry checks WITHOUT building the full project or running the solver — it operates on the block geometry directly.

**Validation checks:**

1. **Overlap warning (last-defined-wins):** For each pair of blocks (i, j) where j > i, check if their AABBs overlap (all 3 axis intervals intersect). If so, append warning:
   `"Warning: Block '{block_j.name}' overlaps '{block_i.name}' — {block_j.name} takes priority in overlap region (last-defined-wins rule)"`

   AABB overlap check: Two blocks overlap if and only if:
   - `a.x < b.x + b.width and b.x < a.x + a.width` (x overlap)
   - `a.y < b.y + b.depth and b.y < a.y + a.depth` (y overlap)
   - `a.z < b.z + b.height and b.z < a.z + a.height` (z overlap)

2. **Powered block thermal path warning:** For each block with power_w > 0, check if it shares at least one face (not just edge/corner) with another non-air block. Face sharing means: one axis has exact boundary alignment (block_a.x + block_a.width == block_b.x, or similar for other faces) AND the remaining two axes have overlapping intervals (not just touching at a point/edge).

   If a powered block does not share any face with another solid block, append:
   `"Warning: Powered block '{name}' ({power_w}W) does not share a face with any other solid block — heat may only dissipate to air"`

3. **Air gap between powered block and metal:** For each powered block, if there exists another block with a typical thermal-path material (any material whose name contains 'Aluminum', 'Copper', 'Steel', or 'Metal', case-insensitive) but they don't share a face, AND there is space between them (gap in any axis), append:
   `"Warning: Powered block '{name}' has gap between it and '{metal_block.name}' — air gap may impede thermal path"`

Place this method in the "Public API" section of BlockEditorWidget, after `build_project()` and `load_project()`.

The method reads blocks via `self._read_blocks()` (already defined, returns list[AssemblyBlock]).

**Integration note:** This method is called externally (e.g., by the main window before solve or by the simulation controller). It does NOT auto-display warnings — it returns strings for the caller to present. This keeps the editor decoupled from the display mechanism.
